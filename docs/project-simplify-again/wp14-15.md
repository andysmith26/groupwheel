## Strategic Context

WP1.4 and WP1.5 complete the math teacher speed path. Where WP1.1–1.3 optimize the _getting to groups_ flow, these two items optimize what happens _after_ groups are on screen and _between_ sessions.

WP1.4 (Pairing History) builds trust — the teacher needs proof that rotation is working. WP1.5 (Projection Mode) makes the live view actually usable in a classroom. Together they close the loop: generate → project → verify rotation → repeat tomorrow with confidence.

## WP1.4: Pairing History Visualization

### Problem

The "avoid recent groupmates" algorithm works, but the teacher has no way to verify it. After 5+ sessions, they need to answer: "Is every student actually working with different people, or are some students always ending up together?" Without this visibility, trust erodes and the teacher reverts to popsicle sticks.

The data infrastructure exists (`getProgramPairingStats`, `getPairingHistory`) but has no UI on the activity detail page — only on a separate analytics route most teachers will never discover.

### Solution

Add a "Rotation Coverage" section to the activity detail page that answers three questions at increasing levels of detail:

1. **Coverage bar** (headline): "What percentage of possible pairings have occurred?"
2. **Per-student partner counts** (scannable list): "How many unique partners has each student had?"
3. **Session timeline** (already exists): "What did past groupings look like?"

The heat map matrix (students × students) is explicitly **out of scope for v1** — it's powerful but complex UI that serves power users, not the daily-use math teacher. The existing `PairingStatsPanel` pair list is also already available on the analytics page for anyone who wants pair-level detail.

### Implementation

#### Part A: Coverage Computation

Coverage is a pure derived metric: the number of unique pairs that have co-occurred in at least one session, divided by the total number of possible pairs.

**No new use case needed.** The `ProgramPairingStatsResult` already returns `pairs` (all pairs with count ≥ 1) and we can derive total possible pairs from the student count.

```typescript
// Pure computation — belongs in a UI utility or computed in the component
function computeCoverage(pairCount: number, studentCount: number): number {
  const totalPossible = (studentCount * (studentCount - 1)) / 2;
  if (totalPossible === 0) return 0;
  return Math.round((pairCount / totalPossible) * 100);
}
```

For a class of 24 students, there are 276 possible unique pairings. After 5 sessions of groups-of-4, roughly 90 pairings will have occurred (~33% coverage). After 15 sessions, coverage might reach 60–80%. This math matters for the explainer text — 33% after 5 sessions is _normal and expected_, not bad.

#### Part B: Per-Student Unique Partner Count

Derive from the same `ProgramPairingStatsResult.pairs` array:

```typescript
function computeUniquePartners(
  pairs: PairingStat[],
  studentCount: number
): Map<string, { name: string; uniquePartners: number; maxPossible: number }> {
  const partnerCounts = new Map<string, { name: string; count: number }>();

  for (const pair of pairs) {
    // Student A
    const a = partnerCounts.get(pair.studentAId) ?? { name: pair.studentAName, count: 0 };
    a.count++;
    partnerCounts.set(pair.studentAId, a);

    // Student B
    const b = partnerCounts.get(pair.studentBId) ?? { name: pair.studentBName, count: 0 };
    b.count++;
    partnerCounts.set(pair.studentBId, b);
  }

  const maxPossible = studentCount - 1;
  const result = new Map();
  for (const [id, { name, count }] of partnerCounts) {
    result.set(id, { name, uniquePartners: count, maxPossible });
  }
  return result;
}
```

##### Data loading on section expand

The rotation section should **lazy-load** its data — don't call `getProgramPairingStats` on every activity detail page visit. Load when the section is first expanded:

```typescript
let rotationExpanded = $state(false);
let pairingStats = $state<ProgramPairingStatsResult | null>(null);
let loadingPairing = $state(false);

async function handleRotationToggle(expanded: boolean) {
  rotationExpanded = expanded;
  if (expanded && !pairingStats && !loadingPairing) {
    loadingPairing = true;
    const result = await getProgramPairingStats(env, { programId: program.id });
    if (isOk(result)) {
      pairingStats = result.value;
    }
    loadingPairing = false;
  }
}
```

The collapsed summary line ("47% of pairings covered · 7 sessions") requires knowing the coverage without loading the full pairing stats. Two options:

**Option A: Always load pairing stats on mount (simple, slightly wasteful)**

Acceptable for now — the query reads from IndexedDB, not a remote server. For a class of 24 across 10 sessions, that's ~240 placements and ~276 pairs. Sub-millisecond.

**Option B: Cache the summary in localStorage (premature optimization)**

Skip this. IndexedDB reads are fast enough.

**Recommendation:** Option A. Load pairing stats alongside the existing session and scenario data on mount. The coverage summary is always visible (when ≥ 2 sessions), so the data must be available without user interaction.

## WP1.5: Classroom Projection Mode

### Problem

The current live view uses standard web typography (`text-2xl` / ~24px group names, `text-lg` / ~18px student names). On a classroom projector viewed from 30 feet, these are unreadable. The page also shows navigation chrome (header, tab bar, "Done" button) that wastes screen space and is irrelevant when projecting.

### Solution

Add a projection mode to the live view that:

1. Enters browser fullscreen (Fullscreen API).
2. Hides all navigation chrome.
3. Scales typography to classroom-readable sizes (48px+ group names, 24px+ student names).
4. Auto-adapts grid columns based on group count to maximize card size.
5. Exits cleanly with ESC.

### Design Decisions

**Fullscreen API vs. CSS-only:** Use the Fullscreen API (`document.documentElement.requestFullscreen()`). CSS-only approaches (hiding nav with a flag) don't reclaim the browser chrome (address bar, tabs), which is significant screen real estate on a projector.

**Where to apply projection mode:** On the `StudentView` component's "All Groups" sub-view only. "Find My Group" search doesn't need projection mode — it's used interactively on the teacher's laptop while students are looking at the projector showing "All Groups".

**Color scheme:** Keep the existing colored group headers (high contrast: white text on colored backgrounds). The card body should be white bg with dark text — this is already what `StudentView` does. No need for a dark mode variant; white backgrounds project better on most classroom projectors.

**Auto-advance (group-at-a-time reveal):** Explicitly **out of scope for v1**. It's a "nice to have" that adds significant complexity (timing controls, pause/resume, presentation state machine) for marginal value. The "All Groups" grid already shows everything at once, which is what most teachers want.

### Implementation

#### 1. Projection mode state on live page

**File:** `src/routes/activities/[id]/live/+page.svelte`

Add state and handlers:

```typescript
let isProjecting = $state(false);

async function enterProjectionMode() {
  try {
    await document.documentElement.requestFullscreen();
    isProjecting = true;
  } catch (e) {
    // Fullscreen denied (e.g., iframe restriction, user denied)
    // Fall back to CSS-only mode
    isProjecting = true;
  }
}

function exitProjectionMode() {
  if (document.fullscreenElement) {
    document.exitFullscreen();
  }
  isProjecting = false;
}

// Sync state when user exits fullscreen via ESC (browser handles ESC → exitFullscreen)
function handleFullscreenChange() {
  if (!document.fullscreenElement && isProjecting) {
    isProjecting = false;
  }
}

onMount(() => {
  document.addEventListener('fullscreenchange', handleFullscreenChange);
  // ... existing onMount logic
});

onDestroy(() => {
  document.removeEventListener('fullscreenchange', handleFullscreenChange);
  // ... existing onDestroy logic
});
```

**c) Auto-scaling grid columns:**

Use a CSS class that adapts columns based on group count. The fewer the groups, the larger each card can be.

```svelte
<style>
  .projection-grid {
    /* Default: auto-fit to fill screen width, minimum 400px per card */
    grid-template-columns: repeat(auto-fit, minmax(400px, 1fr));
  }
</style>
```

This CSS-only approach handles the common cases:

- 2 groups → 2 columns (each ~50% width)
- 4 groups → 2 columns (2 rows of 2)
- 6 groups → 3 columns (2 rows of 3)
- 8+ groups → cards get smaller but `minmax(400px, 1fr)` ensures readability

For classes with many groups (10+), cards will be smaller but the `minmax` constraint keeps them above the readability threshold. If even 400px is too wide for 10+ groups, the grid will scroll vertically — which is acceptable for projection.

An alternative is to compute the grid template in JS based on group count:

```typescript
let projectionCols = $derived.by(() => {
  const count = groupedAssignments.length;
  if (count <= 2) return 2;
  if (count <= 4) return 2;
  if (count <= 6) return 3;
  if (count <= 9) return 3;
  return 4; // 10+ groups
});
```

**Recommendation:** Use the CSS `auto-fit` approach first. It's simpler, responsive, and handles edge cases (resizing, different projector resolutions) automatically. Only switch to JS-computed columns if testing reveals layout problems.

#### 5. Keyboard handling

The existing ESC handler on the live page calls `handleDone()` (archives session, navigates to workspace). In projection mode, ESC should exit projection instead:

```typescript
function handleKeydown(event: KeyboardEvent) {
  if (event.key === 'Escape') {
    if (isProjecting) {
      // Browser already exits fullscreen on ESC; just update state
      // (handled by fullscreenchange listener)
      return;
    }
    handleDone();
  }
}
```

The browser's Fullscreen API already intercepts ESC to exit fullscreen, firing the `fullscreenchange` event which our listener catches and sets `isProjecting = false`. So the ESC handler in the live page only needs to avoid calling `handleDone()` when in projection mode.

## Architecture Notes

### Layer boundaries

Both items are **pure UI work**. No domain, port, infrastructure, or use case changes.

- **WP1.4** creates two new presentational components and wires an existing use case (`getProgramPairingStats`) into the activity detail page. The coverage and unique-partner computations are pure functions derived from existing data — they belong as utility functions or `$derived` computations in the component, not in use cases.
- **WP1.5** adds a visual mode to an existing component. It's entirely CSS + Fullscreen API + conditional rendering. No data changes at all.

### Why coverage isn't a use case

You could argue `computeCoverage` belongs in an application-layer use case. I'd push back: it's a trivial arithmetic formula (`coveredPairs / C(n,2)`) over data that's already returned by `getProgramPairingStats`. Creating a use case for it would be ceremony without value. If the formula ever becomes complex (e.g., weighted by recency, adjusted for students joining mid-semester), then extract it. Until then, keep it as a derived computation.

### Fullscreen API fallback

The Fullscreen API can fail (iframe sandboxing, user permission denied, unsupported browser). The implementation uses a CSS fallback (`fixed inset-0 z-50`) that provides the same visual result minus the browser chrome reclamation. This is acceptable — the navigation chrome (address bar) is a minor screen real estate loss on a projector, and the teacher likely has their browser in fullscreen mode already via F11.
