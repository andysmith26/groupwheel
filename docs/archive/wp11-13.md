**Date:** February 2026
**Status:** Ready for implementation
**Audience:** Engineering team (coding agent + human reviewer)

## Strategic Context

GroupWheel serves a math teacher who forms random-ish groups daily (180+ times/year). The single most important performance metric for this persona is **time from app open to groups projected on screen**. Current flow requires 4+ taps across 3 screens. Target: **< 3 taps, < 10 seconds** for a returning user.

This spec covers three independently shippable work items that together cut the critical path in half.

## WP1.1: One-Tap "Generate & Show" on Activity Detail

### Problem

The "New Groups" card on the activity detail page generates groups and navigates to the workspace. The teacher then must find and tap "Show to Class" to get groups on the projector. This intermediate workspace visit is unnecessary for the daily-use persona вҖ” they don't want to inspect or edit, they want to project.

### Solution

Add a primary action that chains `quickGenerateGroups` вҶ’ `showToClass` вҶ’ `goto('/live')` in a single tap. Demote the current behavior (generate вҶ’ workspace) to a secondary action for teachers who want to inspect first.

### Implementation

#### 1. New handler on activity detail page

**File:** `src/routes/activities/[id]/+page.svelte`

Add a `handleGenerateAndShow` function alongside the existing `handleNewGroups`:

```typescript
async function handleGenerateAndShow() {
	if (!env || !program) return;
	isGeneratingNew = true;
	generateAndShowError = null;

	saveGenerationSettings(program.id, generationSettings);

	// Step 1: Generate groups
	const genResult = await quickGenerateGroups(env, {
		programId: program.id,
		groupSize: generationSettings.groupSize,
		groupNamePrefix: 'Group',
		avoidRecentGroupmates: generationSettings.avoidRecentGroupmates,
		lookbackSessions: generationSettings.lookbackSessions
	});

	if (isErr(genResult)) {
		generateAndShowError = mapGenerateError(genResult.error);
		isGeneratingNew = false;
		return;
	}

	const newScenario = genResult.value;

	// Step 2: Show to class (creates session + placements)
	const showResult = await showToClass(env, {
		programId: program.id,
		scenarioId: newScenario.id
	});

	if (isErr(showResult)) {
		generateAndShowError = `Groups generated but failed to show: ${showResult.error.type}`;
		// Fall back to workspace вҖ” groups exist, just couldn't publish
		goto(`/activities/${program.id}/workspace`);
		return;
	}

	// Step 3: Navigate to live view
	isGeneratingNew = false;
	goto(`/activities/${program.id}/live`);
}
```

Key design decisions:

- If generation succeeds but `showToClass` fails, fall back to workspace (groups exist, teacher can manually "Show to Class" from there). Don't lose the generated groups.
- Show a loading state ("Generating...") during the entire chain since it involves two sequential async operations.

## WP1.2: Single-Activity Auto-Navigation

### Problem

When a teacher has exactly one activity, the dashboard shows a single card. Tapping it is pure friction вҖ” it's the only thing they can do. For the daily-use persona, this costs one tap and ~2 seconds on every visit.

### Solution

On dashboard mount, if the teacher has exactly one activity, redirect to that activity's detail page. Provide an escape hatch so the teacher can always reach the dashboard when they want to (e.g., to create a second activity).

### Implementation

#### 1. Auto-redirect on dashboard

**File:** `src/routes/activities/+page.svelte`

After the activity list loads, check for the single-activity condition:

```typescript
// After activities load:
if (activities.length === 1 && !explicitDashboardVisit) {
	goto(`/activities/${activities[0].program.id}`, { replaceState: true });
	return;
}
```

#### 2. Explicit-visit detection

The redirect must NOT fire when the teacher intentionally navigates to the dashboard (e.g., by clicking "вҶҗ All Activities" breadcrumb, or the nav logo). Two viable approaches:

**Option A: Query parameter (recommended)**

The "вҶҗ All Activities" breadcrumb and any nav links to the dashboard append `?dashboard=true`:

```svelte
<!-- In activity detail page -->
<a href="/activities?dashboard=true">вҶҗ All Activities</a>
```

Dashboard reads this param:

```typescript
const explicitDashboardVisit = $page.url.searchParams.has('dashboard');
```

**Option B: sessionStorage flag**

Set a `sessionStorage` flag when the user navigates to dashboard via an explicit action. Clear it after the dashboard renders.

**Recommendation:** Option A. It's stateless, URL-inspectable, and survives page refreshes without stale state. The `?dashboard=true` param is stripped after render (via `replaceState`) so it doesn't pollute the URL.

#### 3. Breadcrumb update on activity detail page

**File:** `src/routes/activities/[id]/+page.svelte`

Update the "вҶҗ Activities" link to include the escape-hatch param:

```svelte
<a href="/activities?dashboard=true">вҶҗ All Activities</a>
```

#### 4. Edge cases

| Scenario                               | Expected behavior                                                                           |
| -------------------------------------- | ------------------------------------------------------------------------------------------- |
| Zero activities                        | No redirect. Dashboard shows empty state with "Create Activity" CTA.                        |
| Two+ activities                        | No redirect. Dashboard shows all cards.                                                     |
| Activity deleted, leaving one          | Next visit auto-redirects. Correct behavior.                                                |
| Direct URL to `/activities` (no param) | Single-activity user gets redirected. Intentional вҖ” fastest path.                           |
| `replaceState: true`                   | Redirect replaces history entry. Back button skips the dashboard, preventing redirect loop. |

### Acceptance Criteria

1. Teacher with exactly one activity opens `/activities` вҶ’ lands on that activity's detail page automatically.
2. Teacher with 2+ activities sees the dashboard normally.
3. Teacher with 0 activities sees the dashboard empty state.
4. Clicking "вҶҗ All Activities" from the activity detail page always shows the dashboard (no redirect).
5. Browser back button from activity detail goes to the page before the dashboard, not into a redirect loop.

## Architecture Notes

### Layer boundaries

All three items respect hexagonal architecture:

- **WP1.3 Part B** touches algorithm utilities (infrastructure) and threading through application layer (use case input types). No domain changes.
- **WP1.3 Part A** is pure UI. Reads settings from localStorage utility, passes to use case via facade.
- **WP1.1** is pure UI orchestration вҖ” calls two existing facade functions in sequence.
- **WP1.2** is pure UI routing logic.

### No new use cases needed

WP1.1 chains two existing use cases (`quickGenerateGroups` + `showToClass`) at the UI layer. This is intentional вҖ” the UI is the orchestrator for page-level navigation workflows. Creating a composite `generateAndShow` use case would be premature: the two operations have different failure modes and the fallback behavior (navigate to workspace on partial failure) is a UI routing concern, not a business logic concern.

### Settings storage

`generationSettings` uses localStorage (not IndexedDB) intentionally. These are per-device UI preferences, not domain data. They don't need to survive an export/import cycle, and they don't need the transactional guarantees of IndexedDB.
